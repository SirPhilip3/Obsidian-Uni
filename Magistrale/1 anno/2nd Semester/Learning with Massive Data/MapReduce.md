# Reliable Store 

In a distributed setting if a node fails with standard file systems te data is lost 

**Distributed File System** provides a global file namespace 

Data is rarely updated in place 
## Hadoop Distributed File System (HDFS)

**Chuck Servers** : 
+ File is split into contiguous chunks , each of size $16\to64MB$ 
+ Every chunk is replicated , usally 2 to 3 times

**Master Node**
+ Stores the metadata of the whole filesystem
+ Can be replicated for redundancy

A client that wants to retrieve a file talks to the *master* to find the chuck server that contains that file, connects to it directly and retrieves the file 

In *HDFS* we support the current file operations : 
+ *Concurrent reads* 
+ *Single Writer* 
+ Write can only *Append*

## Sequence vs Map Files

**Sequence Files** are a persistent data structure for binary *key-value* pairs where we can :
+ *Write* only with `append()`
+ *Read* only by using `next()`

A **Map File** is a *sorted* *Sequence File* with an *index* to allow for lookups by *key* 

We can *read* by using `next()` and `get()` methods , this is implemented with two sequence files :
+ *Data* : stores key-value pairs *sorted* by *key*
+ *Index* : stores a subset of keys and their position in the *Data* file

>[!note] 
>The *Index* file must be loaded into memory for random look-ups
## MapReduce

Everything is built on top of *key-value* pairs 

>[!note] 
>*keys* and *values* are user-defined , they can be any type

We can use only two *user-defined* functions : 
+ $map(k^1,v^1) \to list(k^2,v^2)$

This given an input pair $(k^1,v^1)$ produces intermediate data $v^2$ *labeled* with a key $k^2$ , there is one map function call for each *key-value* pair in the input 

+ $reduce(k^2,list(v^2)) \to list(k^3,v^3)$ 

Given a key $k^2$ and *associated* data items $list(v^2)$ , produces part of the output of the implemented algorithm as $list(k^3,v^3)$

>[!note] 
>$list(v^2)$ includes all $v^2$ generated by all map function calls 

There is *one* reduce function call for each distinct $k^2$
### Processes

Everything happends in *parallel* 

We have a set of processes names *mappers* and *reducers*

1. Map phase
	1. Performs map functions call
	2. Each mapper processes only a *split* of the input 
2. Shuffle phase
	1. Phase that *transfers* the *data* associated with a given *key from* the *mappers* to the *proper reducer*
	2. It's a parallel group by key
	3. Reducers will recieve data sorted by key 
3. Reduce phase
	1. Each reducer performs *several reduce* function calls 
	2. A reducer produces only a *portion* of the output
	3. Typically each reducer writes to a different files 

>[!example] Word Count
>
>```python
>map(key, value):    # key: document name, value: text of the document 
>	for each word w in value:
>		emit(w,1)
>
>reduce(key, value): # key: a word; value: an iterator over counts
>	result = 0
>	for each count v in values:
>		result += v
>	emit(key,result)
>```

There is a *master node* that takes care of coordination :
+ Task status ( idle , in-progress , completed )
+ Idle tasks get scheduled as workers become available 
+ When a map task completes it sends the master the location and sizes of its $R$ intermediate files , one for each reducer
+ Master pushes this informations to the reducers
+ Master checks periodically for *failures* 
#### Combiners

*Map tasks* on the *same node* will produce many pairs with the same key , we can save network time by pre-aggregating at the *mapper* ( partial aggregation ) :
+ $Combine(k^2,list(v^2)) \to list(k^3,v^3)$

This is executed by exploiting the $reduce$ function for the *reducers*

#### Partition Function

The *partition function* defines which key goes to which reducer 

The default *partition function* is :
$$
hash(key)\ mod\ R
$$
>[!note] 
>This can be overridden and customized
### Failures

**Map worker** failure : 
+ Map tasks *completed* or *in-progress* at that worker are reset to *idle*
+ *Reduce workers* are notified when tasks get rescheduled on another worker

**Reduce worker** failure : 
+ Only *in-progress* tasks are reset to *idle*
+ A different reducer takes the idle task over

**Master** failure : 
+ MapReduce task is aborted 

### Number of Jobs

>[!note]
>$M$ are the number of map tasks
>$R$ are the reduce tasks  

In general make $M$ and $R$ much *larger* than the *number of nodes* in the cluster , since it improves load balancing and speeds up recovery from worker failure but increases overhead

The *number of mappers* is determined by the number of input *splits*

Usally $R$ is *smaller* than $M$ , the system has a *maximum* capacity of *parallel reducers*, therefore they are executed in *waves* : 
1. The shuffling of the first wave is done in parallel with the mappers
2. The shuffling of other waves is done later 

The output of the *reducers* is pread across $R$ files 

### Back-up Tasks

Since **Slow workers** significantly lengthen completion time, near the end of a phase we spawn *backup copies* of tasks , whichever finishes first wins

### Conclusion

MapReduce is not for perfomrance since the shuffling creates huge overheads

It's easy to code the algorithm since we just need to override two functions 

We get *fault tolerance* for *free*

#### Problems hard to implement in MapReduce

**All Pairs Similarity Search** problem : 
+ Given a collection of documents find any pairs of documents with similarity larger than $\tau$ 
+ With $N$ documents we have $N^2$ candidates ( $60MB$ documents generate $40GB$ of intermediate data )

**Graph mining** problems : 
+ Find communities, find leaders, [[PageRank]]
+ We need to share the adjacency matrix between every node 

## MapReduce by Spark 

*Spark* supports massive parallel computations based on **RDD** ( *Resilient Distributed Dataset* ) 

An *RDD* can be seen as a list of data stored on several machines where each machine can access and process part of it 

Spark allows to work on *RDD* with **transformations** and **actions** :
+ *transformations* create a *new* *RDD* from an input *RDD* 
+ *actions* return a list of values to the driver program or to be writte to the file system

>[!important] 
>*Spark* uses lazy evaluation for *transformations* , they are not applied immediatly , they are accumulates until an *action occurs*
